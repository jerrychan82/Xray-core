# Xray-core Drop 协议升级修改计划

## 一、需求背景

### 1.1 业务场景
在直播推流场景中，需要针对特定目标IP段进行网络质量降级处理，以模拟网络不稳定环境。具体需求包括：
- 对特定IP段的上行流量（推流数据）进行概率性丢包
- 对特定IP段的下行流量（拉流数据）进行概率性丢包
- 对特定IP段的所有流量进行概率性丢包
- 丢包率可配置（0-100%）
- 支持与现有路由规则条件组合使用（IP段、用户、端口、入站标签等）

### 1.2 现有方案局限性
- **blackhole协议**：100%丢包，无法实现概率性丢包
- **负载均衡器方案**：需要创建多个出站，配置复杂，精度有限
- **tc方案**：系统层面控制，无法识别Xray内部的路由条件（user、inboundTag等）

### 1.3 目标
在Xray-core中新增一个`drop`出站协议，支持：
- 可配置的随机丢包率（0-100%）
- 支持按方向选择丢包（上行/下行/全部）
- 与现有路由规则完美集成
- 不影响其他出站协议的正常工作

## 二、技术方案设计

### 2.1 整体架构

```
┌─────────────────────────────────────────────────────────────┐
│                     路由规则匹配层                            │
│  ┌──────────────────────────────────────────────────────┐  │
│  │ 匹配条件：IP段、用户、端口、入站标签等                │  │
│  │ 选择出站：drop-uplink-30、drop-downlink-20 等        │  │
│  └──────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│                    Drop 协议处理层                           │
│  ┌──────────────────────────────────────────────────────┐  │
│  │  1. 读取配置：lossPercent、direction                 │  │
│  │  2. 判断数据流方向（上行/下行）                       │  │
│  │  3. 根据direction配置决定是否处理                     │  │
│  │  4. 根据lossPercent随机决定是否丢包                   │  │
│  │  5. 丢包：中断连接或丢弃数据包                        │  │
│  │  6. 不丢包：正常转发（类似freedom协议）               │  │
│  └──────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│                      数据流层                                │
│  上行：客户端 → 服务器（inboundLink.Writer）               │
│  下行：服务器 → 客户端（outboundLink.Writer）               │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 协议配置结构

#### 2.2.1 配置字段说明

| 字段名 | 类型 | 取值范围 | 说明 |
|--------|------|----------|------|
| lossPercent | float64 | 0.0 - 100.0 | 丢包率百分比，0表示不丢包，100表示全部丢包 |
| direction | string | "upload" / "download" / "all" | 丢包方向：upload=仅上行，download=仅下行，all=全部方向 |

#### 2.2.2 配置示例

```json
{
  "protocol": "drop",
  "settings": {
    "lossPercent": 30.0,
    "direction": "upload"
  }
}
```

### 2.3 方向判断逻辑

#### 2.3.1 数据流方向识别

在Xray-core的架构中：
- **上行（Upload）**：数据从客户端流向服务器，对应 `inboundLink.Writer` 或 `link.Writer`（当从inbound角度看）
- **下行（Download）**：数据从服务器流向客户端，对应 `outboundLink.Writer` 或 `link.Reader`（当从outbound角度看）

在drop协议的`Process()`方法中：
- `link.Reader`：读取来自上游的数据（下行方向）
- `link.Writer`：写入到下游的数据（上行方向）

#### 2.3.2 处理逻辑流程

```
开始处理数据流
    ↓
读取配置：lossPercent、direction
    ↓
判断当前数据流方向
    ├─→ 上行（Writer方向）
    └─→ 下行（Reader方向）
    ↓
检查direction配置
    ├─→ "upload"：仅处理上行
    ├─→ "download"：仅处理下行
    └─→ "all"：处理所有方向
    ↓
如果方向匹配，执行丢包判断
    ├─→ 生成随机数（0-100）
    ├─→ 随机数 < lossPercent？
    │   ├─→ 是：执行丢包（中断或丢弃）
    │   └─→ 否：正常转发
    └─→ 如果方向不匹配，正常转发
```

### 2.4 丢包实现方式

#### 2.4.1 丢包策略

**核心原则**：在长连接中随机丢弃数据包，保持连接不断开，模拟真实网络丢包环境。

drop协议参考freedom协议的数据转发机制，实现方式如下：

1. **数据包级别的丢包处理**：
   - 从`link.Reader`读取数据包（`MultiBuffer`）
   - 对每个数据包独立进行随机判断
   - 如果决定丢包：释放该数据包的buffer，不转发，继续处理下一个数据包
   - 如果决定不丢包：正常转发到`link.Writer`或目标连接

2. **TCP连接的处理**：
   - **不中断连接**：连接保持打开状态
   - **数据包丢失**：某些数据包被丢弃后，TCP协议栈会自动触发重传机制
   - **效果**：模拟真实网络中的丢包场景，导致延迟增加、吞吐量下降，但连接不会断开

3. **UDP数据包的处理**：
   - 直接丢弃数据包，不转发
   - UDP本身是无连接的，丢包不会影响后续数据包

4. **双向数据流处理**：
   - **上行方向**：从`link.Reader`读取数据，写入目标连接时进行丢包判断
   - **下行方向**：从目标连接读取数据，写入`link.Writer`时进行丢包判断
   - 每个方向独立判断，互不影响

#### 2.4.2 数据包处理流程

```
读取数据包（MultiBuffer）
    ↓
遍历每个数据包（Buffer）
    ↓
检查方向配置（direction）
    ├─→ 方向匹配？
    │   ├─→ 否：正常转发，继续下一个
    │   └─→ 是：进入丢包判断
    ↓
生成随机数（0-100）
    ↓
随机数 < lossPercent？
    ├─→ 是：释放buffer（丢包），继续下一个
    └─→ 否：正常转发，继续下一个
    ↓
处理完所有数据包
```

#### 2.4.3 随机数生成

使用Go标准库的`math/rand`或`crypto/rand`生成随机数，确保：
- 每个数据包独立判断，互不影响
- 长期统计结果接近配置的丢包率
- 线程安全（如果使用共享随机数生成器）

#### 2.4.4 与真实网络丢包的对比

| 特性 | 真实网络丢包 | drop协议模拟 |
|------|------------|-------------|
| 连接状态 | 保持连接 | 保持连接 ✅ |
| 数据包丢失 | 随机丢失 | 随机丢失 ✅ |
| TCP重传 | 自动触发 | 自动触发 ✅ |
| 延迟增加 | 是 | 是 ✅ |
| 吞吐量下降 | 是 | 是 ✅ |
| 连接断开 | 仅在严重丢包时 | 不会断开 ✅ |

**说明**：drop协议通过数据包级别的随机丢弃，完美模拟了真实网络中的丢包场景。对于TCP长连接（如推流），丢包会触发TCP重传，导致延迟增加和吞吐量下降，但连接保持稳定，这正是模拟"垃圾网络环境"所需要的效果。

## 三、实现方案

### 3.1 文件结构

```
xray-core/
├── proxy/
│   └── drop/
│       ├── drop.go          # 主处理逻辑
│       ├── config.go        # 配置结构定义
│       ├── config.pb.go     # Protobuf生成文件（自动生成）
│       └── config.proto     # Protobuf定义文件
```

### 3.2 核心实现逻辑

#### 3.2.1 协议注册

在`drop.go`的`init()`函数中注册drop协议，使其可以被Xray-core识别和加载。

#### 3.2.2 配置解析

实现`Config`结构体的解析方法，从JSON配置中读取`lossPercent`和`direction`参数。

#### 3.2.3 处理流程

在`Process()`方法中，参考freedom协议的双向数据流处理方式：

**上行方向处理（客户端 → 服务器）**：
1. 从`link.Reader`循环读取数据包（`MultiBuffer`）
2. 遍历每个数据包（`Buffer`）
3. 检查`direction`配置，如果是"upload"或"all"
4. 生成随机数（0-100），判断是否小于`lossPercent`
5. 如果丢包：释放该数据包的buffer，不转发，继续处理下一个
6. 如果不丢包：正常转发到目标连接
7. 连接保持打开，继续处理后续数据包

**下行方向处理（服务器 → 客户端）**：
1. 从目标连接循环读取数据包
2. 遍历每个数据包
3. 检查`direction`配置，如果是"download"或"all"
4. 生成随机数判断是否丢包
5. 如果丢包：释放buffer，不转发
6. 如果不丢包：正常转发到`link.Writer`
7. 连接保持打开，继续处理后续数据包

**关键点**：
- 连接始终保持打开状态，不会中断
- 每个数据包独立判断，互不影响
- TCP丢包会触发自动重传，模拟真实网络环境
- UDP丢包直接丢失，符合UDP特性

#### 3.2.4 方向判断

在`Process()`方法中，通过检查`link`的读写方向来确定数据流方向：
- 写入`link.Writer`的数据 = 上行
- 从`link.Reader`读取的数据 = 下行

### 3.3 与现有架构的集成

#### 3.3.1 路由规则集成

drop协议作为标准的outbound协议，完全兼容现有的路由规则系统：
- 支持所有路由匹配条件（IP段、用户、端口、入站标签等）
- 通过`outboundTag`在路由规则中引用
- 无需修改路由规则的核心逻辑

#### 3.3.2 配置示例

用户可以在路由规则中这样配置：

```json
{
  "routing": {
    "rules": [
      {
        "type": "field",
        "ip": ["1.2.3.0/24"],
        "inboundTag": ["inbound1"],
        "user": ["user1@example.com"],
        "outboundTag": "drop-uplink-30"
      }
    ]
  },
  "outbounds": [
    {
      "tag": "drop-uplink-30",
      "protocol": "drop",
      "settings": {
        "lossPercent": 30.0,
        "direction": "upload"
      }
    }
  ]
}
```

## 五、技术要点

### 5.1 方向判断的准确性

- 需要准确识别数据流方向
- 考虑TCP和UDP的不同处理方式
- 确保在双向数据流中正确应用方向过滤

### 5.2 丢包率的精确性

- 使用高质量的随机数生成器
- 确保长期统计结果接近配置值
- 考虑并发场景下的线程安全

### 5.3 性能影响

- 丢包判断逻辑应尽可能轻量
- 避免影响正常数据转发的性能
- 随机数生成不应成为性能瓶颈

### 5.4 兼容性

- 完全兼容现有Xray-core架构
- 不影响其他出站协议
- 向后兼容现有配置格式
